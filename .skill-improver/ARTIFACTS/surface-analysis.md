## 4. Top 3 Failure Modes Related to Missing Skill Invocations

### Failure Mode 1: Security Testing Is Prose-Only — SE-1/SE-2/SE-3 Findings Are Underpowered

**Root cause:** The `security-testing` skill is never invoked at Phase 0 (SE-1/SE-2/SE-3 attack vectors, lines 315–316) or at Gate 1.5 (adversarial review verification of those same vectors, lines 1086–1098).

**What goes wrong:** Red Team A agents produce Phase 0 security findings by recalling general security knowledge from training data rather than executing a structured STRIDE + OWASP protocol. The `security-testing` skill performs systematic threat modeling. Without it, SE-1 findings are likely to miss macOS-specific attack surfaces (Keychain misuse, entitlement over-granting, sandbox escapes, TCC bypass). At Gate 1.5, Red Team B "verifies" implementation against findings — but without invoking `security-testing`, verification is another prose check. A real injection or path traversal vulnerability surviving to Phase 3 is the concrete failure.

**Impact:** FATAL blocker with no `proved_effective_signoff` reaches Gate 1.5, blocks Phase 3, or — worse — is dismissed without being genuinely verified. Gate 0 and Gate 1.5 may both show PASS while an actual SE-3 vulnerability (path traversal in file open/save operations) ships in the Gate 4 release commit.

---

### Failure Mode 2: /simplify Redefined Inline — Code Quality Degrades Silently

**Root cause:** Hard Rule 19 (line 33) mandates `/simplify` after every file written. The Code Quality Protocol (lines 702–720) redefines the entire simplify pass as a 3-agent inline review (Reuse, Quality, Efficiency). This prose re-implementation means the actual `simplify` Skill tool is never called. The agent executing the inline "simplify" protocol is the same agent that wrote the file — not an independent review.

**What goes wrong:** The `simplify` skill provides an independent, external review with its own methodology. The inline protocol is self-review. The TDD Protocol at line 773 says "Apply `/simplify` to the implementation file" — but without a Skill tool call, this is treated as "run the inline 3-agent description." Over a Medium/Large project with 50+ Swift files, this means 50+ simplify passes that are actually self-reviews. Duplicated logic (`NSRegularExpression` per-call, unconditional `updateNSView` work, duplicated fullscreen toggle) — specifically the "common patterns" listed in lines 714–719 — survives because the agent checking for them is also the one who wrote them.

**Impact:** Gate 2 closes with code that passes build and test but contains hot-path allocations and duplicated logic. Crash check (Hard Rule 21) passes because these are performance/quality issues, not crashes. The app ships with measurable performance regressions under load (SC-1 10k items scenario, SC-2 rapid actions) that only surface post-Gate 4.

---

### Failure Mode 3: TDD Protocol Is Inline — superpowers:test-driven-development Never Activates

**Root cause:** Hard Rule 22 (line 36) mandates TDD. The TDD Protocol (lines 763–775) restates TDD steps verbatim in prose — write test (RED), record in log, implement (GREEN), apply `/simplify`. The `superpowers:test-driven-development` skill is never invoked. The skill's description: "Use when implementing any feature or bugfix, before writing implementation code."

**What goes wrong:** The inline TDD protocol describes the mechanical steps (write test, make it fail, implement, make it pass) but does not activate the full test-driven-development superpower's verification protocol, which includes confirmation that tests fail for the right reason (not just because the function does not exist yet), and that the implementation is minimal (no gold-plating). Agents frequently skip the "confirm RED for the right reason" step when following prose protocols vs an active skill. The result: tests are written that trivially fail (file not found, import missing) rather than failing because the logic is not yet implemented. When implementation is added, these tests pass vacuously without actually testing the acceptance criterion.

**Impact:** Gate 2 closes with G2-3 ("unit tests pass") satisfied, but RTM test cases marked PASSING at Gate 3 do not actually verify their stated acceptance criteria. Specifically: edge cases in Sub-Stage 2A (persistence recovery — SA-2) and Sub-Stage 2B (concurrency — SC-3) have vacuous tests that pass without covering the failure scenarios Red Team A identified in Phase 0. Gate 3 RTM shows all PASS; actual acceptance criteria remain untested in production.
